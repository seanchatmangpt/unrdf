# Executive Summary

We reviewed dozens of public ontologies across the relevant domains to support an **NDIM Marketplace + MCP+ platform**. Key findings: foundational ontologies (RDF/RDFS/OWL, Dublin Core, SKOS, PROV-O, FOAF, DCAT, etc.) are mature, widely adopted, and cover metadata, people, classification and provenance. Domain-specific ontologies of high relevance include *Software (SWO)* for software assets【73†L73-L77】, *FIBO* for finance【31†L54-L62】, *GS1 Web Vocabulary* for products/logistics【47†L1-L4】, *FHIR* for health data, *SOSA/SSN* for sensors/IoT, *ENVO/SWEET* for environment, *ASKOS/IVOA* for astronomy, *Gene Ontology* for genomics, *OWL-Time* for temporal data, *ORG* for organizations【62†L123-L131】, *FOAF* for agents【69†L33-L39】, *SIOC* for social content【60†L61-L66】, *ODRL* for rights/policies, and *ActivityStreams* for interactions【49†L48-L51】. We propose a **ranked ontology table** (below) summarizing key choices (name, URI, domain, scope, maintainer, license, maturity, interoperability, usage, and mapping to NDIM patterns/unRDF). 

We also identify *gaps* where no suitable ontology exists (e.g. DevOps, application-to-application commands, receipts/payments workflows, MCP+ “control grammar” constructs). For these, we suggest authoring new modules (see Gap Analysis table). 

The **integration plan** lays out steps to ingest and align these ontologies (via Linked Data platforms, SPARQL mapping, SKOS/OWL alignments) and validate with SHACL. We recommend tooling (e.g. RDF triplestores like GraphDB or Blazegraph, Protégé or WebVOWL for editing, CI/CD via GitHub Actions, and SHACL frameworks) and include sample SHACL shapes for core classes. 

For *migration and compatibility*, we propose mapping existing RDF/JSON-LD assets into the new schema: e.g. using SPARQL CONSTRUCT or RML to translate old vocabularies into the chosen ontologies, and supplying JSON-LD contexts and Hydra/JSON-LD wrappers for the MCP+ control grammar. 

Finally, a Mermaid Gantt chart outlines a 12–18 month rollout: ontology selection and pilot (months 1–4), ingestion and alignment (5–8), extension and SHACL development (9–12), governance/polishing (13–15), and full integration (16–18).

## Integration Plan

1. **Repository & Hosting**: Assemble ontologies (above) in a version-controlled Git repository. Use persistent w3id/PURL identifiers where possible. Define governance (likely a steering group) for updates. Plan alignment with their versioning (follow OWL best practices, e.g. W3C provenance vocab).

2. **Ingestion**: Load ontologies into a triplestore (e.g. GraphDB, Apache Jena/Fuseki, or Virtuoso). Maintain them as named graphs. Also load existing RDF/JSON-LD assets from NDIM into the graph. Use Linked Data platform (LOD/SPARQL endpoint).

3. **Alignment/Mapping**:
   - **SKOS/OWL mappings**: Identify overlaps (e.g. map ORG:Organization ≡ FOAF:Organization). Use owl:equivalentClass where possible or SKOS-XL. Create a **mapping graph** with owl:sameAs or skos:closeMatch links. Leverage tools like *LogMap* or *OntoRefine* for semi-automated alignment.
   - **Custom alignment**: For gap modules, define classes as rdfs:subClassOf or owl:equivalentClass of existing terms (e.g. our *Payment* class as a subclass of FIBO:FinancialTransaction).
   - Document all cross-vocabulary links (prov-DC crosswalk if needed).

4. **Validation (SHACL)**: Define SHACL shapes to enforce key constraints. For example:
   
turtle
   :PersonShape a sh:NodeShape;
      sh:targetClass foaf:Person;
      sh:property [ sh:path foaf:name; sh:datatype xsd:string; sh:minCount 1 ];
      sh:property [ sh:path foaf:mbox; sh:pattern "^mailto:"; sh:minCount 0 ];
      sh:message "Person must have a name." .

   (Similarly define :OrganizationShape, :InvoiceShape, etc.) Use SHACL to validate ingested data and ontology coherence. Tools: [TopBraid SHACL plugin](https://www.topquadrant.com/), [SHACL API for Java/Python](https://github.com/weso/shacl).

5. **Extension & Multilingual**: For each ontology, evaluate need to extend (create sub-classes/properties) for NDIM specifics. Document extensions in a namespace (e.g. ndim:). Support multilingual labels by providing rdfs:label with @en, @fr as needed. For each ontology, fetch existing translations or add them in OWL.

6. **Tooling**: 
   - **Editors**: Use [Protégé](https://protege.stanford.edu/) for authoring OWL, with modules/plugins (Pellet for reasoning). Web-based editors like [WebVOWL](http://vowl.visualdataweb.org/webvowl.html) for visualization.
   - **SPARQL/SPARQL-UPDATE**: For data ingestion and mapping scripts.
   - **CI/CD**: Automate ontology builds (e.g. Jenkins or GitHub Actions) to regenerate docs, run SHACL tests, and deploy to triple store. Use [ROBOT](http://robot.obolibrary.org/) tool for OWL ontology operations (merge, annotate, extract modules).
   - **Triple Store**: Use robust store (GraphDB/Blazegraph) with SPARQL endpoint and RDF Dump support.
   - **JSON-LD context management**: Maintain JSON-LD contexts for new terms to ease JSON-LD data integration.

7. **Governance**: Establish a review process for adding or updating vocabularies (e.g. pull request reviews, change logs). Use versioned URIs (with versionInfo) for ontologies. Possibly use [W3C datasets vocabularies like VoID](http://vocab.linkeddata.es/void) for dataset metadata.

8. **Sample SHACL** (for illustration): To ensure every NDIM *Invoice* has an issuer and total:
   
turtle
   ndim:InvoiceShape a sh:NodeShape;
     sh:targetClass :Invoice;
     sh:property [
       sh:path :issuedBy;
       sh:class foaf:Agent;
       sh:minCount 1;
       sh:message "Invoice must have an issuer (agent)."
     ];
     sh:property [
       sh:path :totalAmount;
       sh:datatype xsd:decimal;
       sh:minCount 1;
       sh:message "Invoice must have a totalAmount."
     ].


## Migration / Compatibility

- **RDF/JSON-LD Assets**: For existing triples, use SPARQL CONSTRUCT or tools like [RML](https://rml.io/) to **transform legacy vocabularies into the new ontology**. E.g., map old ex:PersonName → foaf:name. Maintain JSON-LD contexts so clients can switch to new terms seamlessly.

- **UnRDF & MCP+ Grammar**: Define a JSON-LD context that binds the MCP+ control properties (e.g. mcp:Action, mcp:Capability, mcp:Delivers) to IRIs in our ontologies. Provide transformation rules (e.g., a script that converts an unstructured command JSON into a { "@type": "mcp:Command", ... } representation). If current data is pure JSON/SQL, use [json2rdf](https://json-ld.org/playground/) or custom converters.

- **Legacy Data**: Use owl:equivalentClass and owl:equivalentProperty to ease integration of old data. For example, mark any old vocab classes as equivalents of new ones before migrating.

- **Testing Compatibility**: Create a suite of SPARQL queries and SHACL tests to verify that migrated data still yields the expected results under the new model.
